# System Overview

---

## **Overview**

The system overview describes how the different components of the lab environment work together to support the SOC alert automation pipeline.

Rather than focusing on a single tool or workflow, this section explains how data moves across systems, how each component contributes to the overall process, and how the environment is structured to simulate a realistic security operations workflow.

The goal is to show that the project is not just a standalone automation, but part of a larger ecosystem that includes detection, data collection, analysis, communication, and incident tracking.

---

## **System Purpose**

The purpose of the system is to create an end-to-end environment where alerts can be generated, processed, enriched, and tracked in a way that reflects real-world SOC operations.

In many lab setups, tools are configured in isolation without demonstrating how they interact. This system is intentionally designed to connect each component so that alerts move through a complete lifecycle, from detection to investigation.

By doing this, the project demonstrates how multiple technologies can be integrated into a cohesive workflow rather than existing as separate, disconnected tools.

---

## **Core Components**

The environment is composed of several core systems that each serve a specific role within the pipeline.

These components include the SIEM platform, endpoint agents, the workflow automation engine, the email notification system, and the ticketing system. Each one contributes a necessary function that supports the overall alert handling process.

Together, these systems form a complete pipeline that allows simulated attack activity to be detected, processed, communicated, and tracked in a structured way.

---

## **Elasticsearch and Kibana (SIEM Layer)**

Elasticsearch and Kibana act as the central SIEM layer within the environment.

Elasticsearch is responsible for storing and indexing all incoming telemetry, allowing large volumes of log data to be searched and analyzed efficiently. It serves as the backend where all alert data is stored after being collected from endpoints.

Kibana provides the interface used to interact with this data. It allows detection rules to be created, alerts to be viewed, and investigations to be performed through dashboards and search queries.

Together, these tools provide the foundation for detection and alert generation within the system.

---

## **Elastic Agent and Fleet (Data Collection Layer)**

Elastic Agents are deployed on endpoint systems to collect telemetry and send it to Elasticsearch.

These agents are managed through Fleet, which acts as a centralized control plane for configuration and policy management. Fleet ensures that all agents are consistently configured and collecting the correct data sources across the environment.

This layer is responsible for generating the raw data that eventually becomes alerts, making it a critical part of the pipeline.

---

## **Detection Layer**

The detection layer is implemented within Kibana using predefined or custom detection rules.

These rules analyze incoming telemetry and identify patterns that may indicate suspicious or malicious activity. When a rule is triggered, an alert is generated and stored in Elasticsearch.

This is the starting point of the automation pipeline, as it produces the alert data that will later be retrieved and processed by n8n.

---

## **n8n (Automation Layer)**

n8n serves as the automation engine that processes alerts after they are generated by the SIEM.

It retrieves alert data from Elasticsearch, extracts the relevant fields, enriches the alert with AI-generated analysis, and coordinates the delivery of notifications and ticket creation.

This component is responsible for transforming raw alert data into structured outputs that are easier to consume and act upon.

---

## **AI Processing Layer**

The AI processing layer is integrated within n8n using a language model to analyze alert data.

This layer takes normalized alert fields and generates a human-readable summary or explanation of the alert. The goal is to provide additional context that can assist analysts during triage.

By incorporating AI into the workflow, the system demonstrates how alert interpretation can be partially automated to improve efficiency and readability.

---

## **Gmail (Notification Layer)**

Gmail is used as the notification layer to deliver alerts in a readable format.

Once the alert has been normalized and enriched, it is formatted into an email and sent to a designated inbox. This ensures that alerts can be quickly reviewed without requiring direct access to the SIEM or automation platform.

This layer improves visibility and allows alerts to be communicated through a familiar and accessible channel.

---

## **osTicket (Ticketing Layer)**

osTicket is used as the ticketing system to track alerts as formal cases.

After the email is sent, the workflow creates a ticket containing the alert details and AI-generated analysis. This ensures that each alert can be documented, assigned, and followed through as part of an investigation process.

This layer adds structure and accountability to the workflow by turning alerts into trackable records.

---

## **Data Flow Across the System**

Data flows through the system in a structured sequence that connects each layer together.

Telemetry is first generated on endpoint systems and collected by Elastic Agents. That data is sent to Elasticsearch, where detection rules analyze it and generate alerts.

n8n then retrieves those alerts, processes and enriches them, sends a notification email through Gmail, and creates a ticket in osTicket for tracking.

This flow ensures that data moves logically from collection to detection to response without breaking continuity between systems.

---

## **System Integration**

The system is designed so that each component interacts through well-defined interfaces rather than operating in isolation.

Elasticsearch exposes alert data through APIs that n8n can query. n8n then acts as the central orchestrator, connecting the SIEM to external systems like Gmail and osTicket.

This integration approach reflects how real-world systems are connected, where APIs and automation tools are used to bridge gaps between platforms.

---

## **Operational Value**

From an operational perspective, the system reduces the amount of manual work required to handle alerts.

Instead of requiring an analyst to manually retrieve alert data, interpret it, send notifications, and create tickets, the system automates those steps while maintaining consistency and structure.

This improves efficiency and allows analysts to focus more on investigation and decision-making rather than repetitive tasks.

---

## **Scalability Considerations**

The system is designed in a way that allows it to scale as additional data sources, detection rules, or automation steps are added.

Because each component has a defined role, new integrations or enhancements can be introduced without disrupting the overall workflow. For example, additional notification channels or enrichment steps could be added within n8n without changing how alerts are generated in Elasticsearch.

This modular design supports future growth and experimentation within the lab.

---

## **Security Engineering Perspective**

From a security engineering perspective, the system demonstrates how multiple tools can be combined to support a complete alert lifecycle.

It shows how detection logic, data processing, automation, and case management can be integrated into a single workflow that improves both efficiency and clarity.

This reflects real-world engineering work, where the goal is not just to deploy tools, but to connect them in a way that improves operational outcomes.

---

## **Documentation Goal**

The goal of this system overview is to provide a clear explanation of how all components in the environment interact and contribute to the overall workflow.

It is intended to help the reader understand how data moves through the system, what each component is responsible for, and why the architecture was designed this way.

Together, this section helps position the project as a complete system rather than a collection of individual tools.
